# 多对文字替换 — 设计文档

## 概述

将替换工具的输入方式从"单对文字"扩展为"动态行列表"，一次执行多对顺序替换。

## UI 变更

| 原 UI | 新 UI |
|---|---|
| 两个单行输入框（查找 / 替换为） | 动态行列表，每行一对输入框 |
| `开始替换` 按钮在右侧 | 按钮移到列表下方 |
| 无可删除操作 | 每行右侧 × 删除按钮 |
| 无排序 | ↑↓ 箭头按钮调整顺序 |

### 行列表区域

- 每行：`查找输入框` + `→` 标签 + `替换输入框` + `✕ 删除按钮` + `↑ ↓ 排序按钮`
- 列表至少保留 1 行（最后一行不显示删除按钮）
- 底部 `+ 添加替换对` 按钮追加空行
- 每行间用细线或微间距分隔

### 窗口调整

- 高度从 580 适当增加到 650，容纳多行输入

## 替换逻辑

### 顺序执行

按列表从上到下的顺序，对每个文件依次执行替换：

```
for replace_pair in pairs:
    replace_in_file(filepath, pair.find, pair.replace)
```

### 已被替换内容不再参与后续匹配

关键规则：**同一个文本位置只会被替换一次**。

实现方式：

- Word/PPT：利用 COM 或 python-docx 的 run 级别替换后，被替换的 run text 已变为新文字，自然不会匹配后续对
- Excel：逐格替换后，格值已变为新文字，后续对不会再次匹配
- 文本文件：先读全量，按顺序对内容逐一 replace，写入结果

对于文本文件，这是自然结果：

```python
content = file_content
for pair in pairs:
    content = content.replace(pair.find, pair.replace)
```

因为 `str.replace` 每次产生新字符串，替换后的文字（如 `2026`）不会再匹配到 `2025->2026`。

### 空行跳过

空查找字符串的行自动忽略，不影响其他行执行。

## 实现方案

### 不新建文件，只修改 `word_replace.py`

改动点：

1. **新增 `ReplacementPair` 数据类** — 封装 find/replace/active
2. **新增 `PairListFrame` 类** — 管理行列表 UI（滚动区域、添加/删除/排序）
3. **修改 `App.__init__`** — 替换单行输入框为 PairListFrame
4. **修改 `App._run`** — 遍历 pair 列表，每个文件按顺序执行多对替换
5. **修改 `process_folder`** — 接受 `[(find, replace), ...]` 列表参数
6. **各 `replace_in_*` 函数不变** — 仍然每次处理一对，由上层循环调用

### 不涉及

- 不改动 bat 文件、requirements.txt
- 不改动 exe 构建流程
- 不新增文件

## 边界情况

- 空查找：跳过该行
- 查找==替换：跳过该行（提示或忽略）
- 全部行为空：不执行（弹出提示）
- 某行替换失败：记录错误，继续执行后续行和后续文件
- 大文件（40MB+）：流式读写的文本类文件性能不变；COM 类文件（.doc/.xls）性能取决于 Word/Excel 响应
